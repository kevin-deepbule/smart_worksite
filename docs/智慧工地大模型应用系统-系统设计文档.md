> **本章总结：** 本文档是在完整阅读仓库全部项目自有 Markdown 文档，并进一步核对 Java、Vue、Python、Flyway、Docker Compose 与运行配置后形成的系统级设计基线。它不是单纯复述需求，也不把规划能力伪装成已实现能力，而是同时描述系统目标、当前实现、强制架构约束、关键业务流程、数据与接口设计、可靠性机制、部署方式及演进路线。若本文档与早期目标文档存在差异，应先区分“现行代码事实”和“目标设计”；当前接口事实以 Controller、迁移和模块 API 文档为准，长期边界以根 `AGENTS.md` 的强制约束为准。

## 1. 系统概述

> **本章总结：** 智慧工地大模型应用系统不是一个单一聊天机器人，而是一套以工地项目为隔离边界、以文件和知识资产为数据基础、以任务编排为执行骨架、以 AI 服务为智能能力来源的企业应用平台。系统将知识问答、合规审查、报告生成、OCR 和政策资讯等场景统一到一个安全、可追踪、可失败恢复的业务闭环中：前端只面向 Java API，Java 掌握身份、权限、项目、事务、状态、文件和审计，Python 封装模型、Agent、RAG、Embedding、rerank、OCR 与网页内容抽取。

### 1.1 业务背景

建筑工地资料通常分散在 Word、PDF、Excel、图片、业务数据库和互联网政策网站中。传统流程存在以下问题：

- 资料查找依赖个人经验，跨文件检索效率低。
- 规范和模板多，合规审查重复、耗时且难留痕。
- 周报、月报和专项报告内容汇总依赖人工复制。
- 证照、车牌、发票和合同字段录入成本高。
- 多项目并行时容易发生数据串用和权限越界。
- 模型调用如果没有状态、引用、错误和审计，难以用于正式业务。

系统建设重点因此不是“接入一个大模型”，而是建立资料接入、结构化处理、知识检索、模型生成、业务持久化、权限隔离和审计追踪的一体化平台。

### 1.2 建设目标

| 目标 | 设计落实 |
| --- | --- |
| 统一智能应用入口 | 工作台统一进入问答、审查、报告和 OCR |
| 多项目隔离 | 业务表使用 `project_id`，应用服务统一校验项目访问与可写状态 |
| 多源知识接入 | 文件、知识库、政策文章和业务数据源统一由 Java 编排 |
| AI 能力解耦 | Python 服务封装模型、Agent、RAG、Embedding、rerank、OCR |
| 长任务可靠执行 | MySQL 任务与 Outbox 为事实源，Redis 为投递通道，Worker 带租约执行 |
| 结果可追溯 | 保存引用、规则证据、变量值、阶段日志、外部调用日志和 requestId |
| 企业级安全 | JWT、RBAC、项目成员校验、AES-GCM、临时文件 URL、敏感日志脱敏 |
| 可演进 | Java 先采用模块化单体，保持未来拆分服务的领域边界 |

## 2. 架构原则、约束与关键决策

> **本章总结：** 系统的首要架构原则是“业务控制权在 Java、智能算法能力在 Python、交互只经由 Java、数据按项目隔离、耗时操作必须可追踪”。这些原则不是代码风格偏好，而是安全、事务一致性、可观测性和团队协作的边界。所有新增能力都应服从模块化单体分层、外部能力适配、显式失败、写入效果确认、逻辑删除和异步任务可靠投递等约束。

### 2.1 关键架构决策

| 编号 | 决策 | 原因 | 影响 |
| --- | --- | --- | --- |
| ADR-01 | Java 主系统采用模块化单体 | 降低一期部署与事务复杂度 | 模块内部独立分层，禁止跨模块 Mapper |
| ADR-02 | Python 作为智能算法服务 | 便于接入模型、RAG、向量库和 AI 生态 | Java 通过 REST 适配器调用 |
| ADR-03 | 前端只调用 Java | 统一鉴权、审计、项目隔离和错误处理 | 禁止直连 Python、MinIO、数据库、向量库 |
| ADR-04 | MySQL 是业务状态事实源 | 业务状态需要事务、一致性和可查询性 | Redis 不得成为唯一任务状态 |
| ADR-05 | 任务采用 Outbox + Redis | 解决数据库提交与消息投递双写问题 | `task_outbox` 持久化后再异步投递 |
| ADR-06 | 文件内容存 MinIO，元数据存 MySQL | 避免大对象占用业务库 | 通过 `file_id` 关联业务 |
| ADR-07 | 知识切片和向量只由 Python 管理 | 避免 Java 耦合 Embedding 与向量引擎 | Java 只提交解析内容和查询条件 |
| ADR-08 | 数据库问答由 Python 规划、Java 执行 | 同时获得自然语言能力和数据库安全控制 | Java 严格验证只读 SQL |
| ADR-09 | 审查按“块证据—规则归约—最终摘要”三层执行 | 避免单块无证据误判全文 | Java 确定总体三态结论 |
| ADR-10 | 报告按变量快照逐项生成 | 支持进度、失败定位和断点续跑 | 每个变量即时持久化 |

### 2.2 强制分层

```mermaid
flowchart TD
    C[Controller<br/>HTTP参数、Bean Validation、响应包装]
    A[Application Service / Facade<br/>用例编排、事务、权限、状态]
    D[Domain<br/>领域对象、枚举、状态转换、规则]
    R[Repository<br/>业务持久化接口]
    M[MyBatis Mapper + XML<br/>SQL与结果映射]
    I[Infra Adapter<br/>Redis、MinIO、Python、JDBC]
    DB[(MySQL)]
    EXT[(外部依赖)]

    C --> A
    A --> D
    A --> R
    A --> I
    R --> M
    M --> DB
    I --> EXT
```

禁止的调用包括：

- Controller 直接调用 Mapper、Redis、MinIO、Python 或数据库。
- 一个业务模块直接调用另一个模块的 Mapper。
- 前端直接访问 Python、数据库、MinIO、向量库或模型。
- Java Controller/Application Service 编写模型 Prompt、Embedding、OCR 或向量检索核心算法。
- Worker 依赖请求线程 `SecurityContext`。
- 用默认内容、空列表或假成功隐藏外部服务、存储或持久化失败。

### 2.3 项目隔离与可写性

所有项目级能力都必须经过 `ProjectAccessApplicationService`：

| 场景 | 入口 |
| --- | --- |
| 用户读 | `requireProjectAccess` |
| 用户管理 | `requireProjectManage` |
| 用户写 | `requireProjectWritableAccess` / `requireProjectWritableManage` |
| 后台 Worker 写 | `requireProjectWritableForSystem` |

项目状态规则：

```mermaid

    DISABLED --> DELETED: 逻辑删除
```

### 2.4 显式失败与写入确认

关键写入必须检查：

- `insert/update/delete` 影响行数。
- 自增 ID 是否生成；必要时按业务键读回。
- API 返回持久化对象前是否能从数据库重新读取。
- 外部调用成功后所需的调用日志是否成功写入。
- 失败状态、阶段日志和 Outbox 失败原因是否成功写入。

零行更新意味着记录缺失、状态已变化或并发冲突，应返回 `40900`，不得报告成功。

### 2.5 当前实现偏差

| 偏差 | 当前事实 | 目标收敛 |
| --- | --- | --- |
| 文件解析 AI 边界 | `file` 模块仍有 `QwenVlDocumentParseAdapter` 和 Java 侧 Qwen-VL 配置 | 改为 Java 调用 `python-ai-service` 文档解析 API；Qwen Key 只留在 Python |
| OCR 任务执行 | OCR 使用 `@Async("ocrTaskExecutor")`，直接更新任务表 | 改为写 `task_outbox`，由统一 Worker claim、租约、心跳和 owner 校验执行 |
| 文件解析任务执行 | 文件解析使用本地 `@Async`，不是统一 Outbox | 逐步纳入统一任务或可靠解析事件链 |
| 生产部署 | Compose 仅包含 MySQL、Redis、MinIO | 补齐前端、Java、Python、网关、向量库、监控与备份 |
| 旧文档状态 | 部分需求表仍将已实现能力标为规划中 | 以 Controller、README、模块 API 和本文档为现行事实 |

## 3. 总体架构

> **本章总结：** 系统采用清晰的三服务边界：Vue 负责交互，Java 负责企业业务和系统控制，Python 负责智能算法。MySQL 保存业务事实，MinIO 保存文件与大文本对象，Redis 负责缓存、令牌安全和任务投递，向量存储由 Python 独占访问。该结构在当前阶段以模块化单体获得较低运维复杂度，同时通过领域模块、适配器和消息边界为后续服务拆分保留空间。

### 4.4 通用响应和错误

业务 JSON 响应：

```json
{
  "code": 0,
  "message": "success",
  "data": {},
  "requestId": "request-id",
  "timestamp": "2026-07-23T10:00:00+08:00"
}
```

现行错误码：

| code | 含义 |
| --- | --- |
| `40000` | 参数错误 |
| `40100` | 未认证 |
| `40300` | 无权限 |
| `40400` | 资源不存在 |
| `40900` | 状态或并发冲突 |
| `42900` | 请求过频或账户临时锁定 |
| `50000` | 系统错误 |
| `50200` | 外部服务异常 |
| `50300` | 服务不可用 |

现行实现对大多数 `BusinessException`、认证和授权错误使用 HTTP 200 加业务 `code`；Bean Validation 使用 HTTP 400，未捕获异常使用 HTTP 500。前端必须同时处理 HTTP 状态和业务 `code`。

## 6. Python 智能算法服务设计

> **本章总结：** Python 服务是统一智能能力边界，使用 FastAPI 暴露稳定的内部 REST 契约，在其内部封装 Qwen、RAG、向量存储、rerank、Agent 工具、OCR 和政策网页抽取。Java 不感知供应商请求格式，只传业务上下文并接收规范化结果。所有模型密钥应只存在于 Python 运行环境，Python 不写 Java 的业务数据库、MinIO 元数据或任务状态。

## 7. 核心业务流程设计

> **本章总结：** 系统的业务闭环都遵循相同模式：用户在前端发起业务用例，Java 完成权限和项目校验、创建可追踪记录，必要时把耗时处理交给异步执行器，Python 只返回智能结果，Java 校验并持久化结果，前端通过业务状态或任务阶段展示进度。以下流程特别强调每一步的事实来源、失败路径和不可绕过的校验。

### 7.1 登录与认证流程

```mermaid
sequenceDiagram
    participant U as 用户
    participant FE as Vue
    participant J as AuthApplicationService
    participant DB as MySQL
    participant R as Redis

    U->>FE: 用户名 + 密码
    FE->>J: POST /api/auth/login
    J->>R: 检查临时锁定
    J->>DB: 查询启用用户、角色、权限、项目
    alt 密码错误
        J->>R: 累加失败计数/必要时加锁
        J-->>FE: 认证失败或 42900
    else 成功
        J->>R: 清理失败计数与锁
        J->>DB: 更新最后登录时间并检查影响行数
        J-->>FE: JWT + 用户/权限/项目
    end
```

后续每次请求都会验签、检查 JWT 黑名单并回查用户当前状态，已停用或删除用户不能依赖旧 Token 继续访问。

## 8. 异步任务与可靠性设计

> **本章总结：** 任务系统以 `generate_task` 表记录业务执行状态，以 `task_outbox` 解决事务提交后可靠投递问题，以 Redis List 作为轻量交付通道，以 Worker claim、租约、心跳和 owner 校验防止重复或过期执行。重试、取消、阶段日志和失败记录都必须可观察。当前报告、知识入库、政策采集和长文档审查已经走该统一路径，文件解析与 OCR 仍保留本地异步路径。

### 8.1 任务类型

统一 Worker 当前支持：

```text
REPORT_GENERATION
KNOWLEDGE_INDEXING
POLICY_CRAWL
REVIEW_EXECUTION
```

其他任务记录还包括：

```text
FILE_PARSE
OCR_RECOGNITION
```

但后两者当前分别通过本地 `@Async` 执行，尚未完全纳入统一队列。

### 8.2 任务状态机

```mermaid
stateDiagram-v2
    [*] --> PENDING
    PENDING --> QUEUED: Outbox准备完成
    QUEUED --> RUNNING: Worker claim
    RUNNING --> SUCCESS: owner完成
    RUNNING --> FAILED: owner失败
    RUNNING --> CANCELED: 检测cancel_requested
    FAILED --> QUEUED: 手动重试且未超限
    PENDING --> CANCELED: 取消
    QUEUED --> CANCELED: 取消
    RETRYING --> QUEUED: 重新投递
    RETRYING --> CANCELED: 取消
    SUCCESS --> [*]
    FAILED --> [*]
    CANCELED --> [*]
```

### 8.3 Outbox 投递

```mermaid
sequenceDiagram
    participant B as Business Transaction
    participant DB as MySQL
    participant D as Outbox Dispatcher
    participant R as Redis
    participant W as Worker

    B->>DB: 写业务记录
    B->>DB: 写 generate_task(QUEUED)
    B->>DB: 写 task_outbox(PENDING)
    DB-->>B: 同事务提交
    D->>DB: 查询到期 Outbox
    D->>R: RPUSH task queue
    alt Redis 投递成功
        D->>DB: Outbox DELIVERED
    else 投递失败
        D->>DB: delivery_count + error + next_delivery_at
        Note over D,DB: 指数退避，最多 5 次
    end
    R->>W: LPOP/BLPOP 消息
```

Outbox 唯一键为 `task_id + event_type + deleted`，防止同一任务事件重复创建。

### 8.4 Worker claim 与租约

```mermaid
sequenceDiagram
    participant W1 as Worker A
    participant W2 as Worker B
    participant DB as MySQL

    W1->>DB: claim QUEUED task
    W2->>DB: claim 同一 task
    DB-->>W1: 更新 1 行，写 worker_id/lease
    DB-->>W2: 更新 0 行，不可领取
    W1->>DB: heartbeat/renewLease
    W1->>DB: completeSuccess/Failure with owner
    alt owner、状态、租约不匹配
        DB-->>W1: 0 行
        W1-->>W1: 409 冲突，不得静默完成
    end
```

### 8.5 重试与取消

重试：

- 仅 `FAILED` 且 `retry_count < max_retry_count`。
- 更新任务为 `QUEUED`，清理错误和取消标记。
- 写 `RETRY_REQUESTED` 阶段日志。
- 新建可投递 Outbox 事件。
- 业务模块必须实现幂等或按阶段恢复。

取消：

- `PENDING/QUEUED/RETRYING` 直接置 `CANCELED`。
- `RUNNING` 只写 `cancel_requested=true`。
- Worker 在阶段边界检查取消；长文档审查在每个块模型调用前续租并检查。
- 终态取消返回冲突。

### 8.6 阶段日志

`task_stage_log` 记录：

```text
task_id
attempt_no
stage_code
status
input_summary
output_summary
error_message
started_at
finished_at
cost_ms
```

阶段日志写入失败视为可观测性破坏，必须使相关操作失败。

### 8.7 本地异步路径风险

文件解析和 OCR 的本地线程池路径存在以下限制：

- 进程退出可能丢失内存中的待执行任务。
- 没有统一 claim、租约和 owner 校验。
- 水平扩容时重复执行控制较弱。
- OCR 对任务写入的影响行数检查不如统一 Worker 完整。

收敛建议：

1. 增加 `FILE_PARSE` 和 `OCR_RECOGNITION` 的统一 Worker 分派。
2. 创建记录与 Outbox 同事务。
3. Worker 使用 system-safe 文件访问。
4. 统一阶段、续租、取消和失败持久化。

### 9.7 数据建模规则

- 业务表默认使用 `id`、`created_at`、`updated_at`、`created_by`、`updated_by`、`deleted`。
- 项目级表保存 `project_id`。
- 查询默认过滤 `deleted = 0`。
- 不依赖数据库外键表达全部领域约束，项目归属和状态由应用层检查。
- JSON 必须由应用层校验并序列化，通过普通 MyBatis 参数写入；禁止 Mapper 使用 `CAST(? AS JSON)`。
- 不能把密码、Token、密钥、永久公开 URL 存入 metadata。
- 数据源密码以 `AES_GCM:` 前缀密文保存，API 不返回明文或密文。

## 11. 安全设计

> **本章总结：** 系统安全采用“认证、平台权限、项目成员、资源归属、资源状态”多层校验。JWT 只解决身份携带，真正授权仍由方法级权限和应用服务项目校验完成。Redis 支持 Token 黑名单和登录失败锁定，数据源密码使用 AES-GCM，文件通过短期 URL 访问，数据库问答只允许有限只读 SQL，AI 与审计日志必须脱敏。任何后台 Worker 都不能借用用户请求线程的认证上下文。

### 11.1 认证链

```mermaid
flowchart LR
    REQ[请求] --> JWT[JWT签名/过期]
    JWT --> BL[Redis黑名单]
    BL --> USER[回查用户未删除且ENABLED]
    USER --> ROLE[角色/权限注入]
    ROLE --> METHOD[@PreAuthorize]
    METHOD --> PROJECT[项目成员/平台管理员]
    PROJECT --> RESOURCE[资源归属与状态]
```

### 11.2 登录安全

- 密码使用 BCrypt。
- 登录失败在 Redis 窗口内计数。
- 默认达到 5 次失败后锁定 900 秒，均可配置。
- 登录成功清理失败计数和锁。
- 退出登录把当前 JWT `jti` 加入 Redis 黑名单。
- JWT 默认有效期 7200 秒。
- 本地 `admin/admin123` 仅用于开发，生产必须重置或禁用。

### 11.3 RBAC 与项目权限

平台角色决定全局权限，项目成员关系决定项目范围，同一用户可在多个项目拥有不同项目角色。

资源访问必须同时满足：

1. 已认证。
2. 拥有接口所需权限。
3. 是项目成员或平台管理员。
4. 资源属于目标项目。
5. 写操作时项目为 `ENABLED`。
6. 资源自身处于允许状态。

### 11.4 文件安全

- Bucket 默认私有。
- Java 授权后生成短期预签名 URL。
- 不返回 MinIO Secret、内部对象名或永久公开地址。
- 预签名下载请求不附加应用 JWT。
- 文件对象名使用项目、业务类型、日期和 UUID，避免原文件名碰撞和信息泄露。

### 11.5 数据源安全

- 密码保存为 `AES_GCM:<base64 payload>`。
- 密钥长度为 16、24 或 32 字节，或 `base64:` 编码。
- 响应不返回密码明文或密文。
- 连接测试和 Schema 使用真实 JDBC。
- 数据库问答连接设置只读、查询超时和最大行数。
- SQL 只允许 `SELECT/WITH`，禁止危险关键字和分号多语句。
- 生产数据源应使用最小权限只读账户和表/字段白名单。

### 11.6 AI 与服务间安全

- Java 调 Python 使用独立服务密钥。
- Qwen Key 只在 Python 环境中。
- 模型请求摘要不保存完整敏感文档。
- OCR 日志不保存图片 base64、身份证号明文、详细地址、临时 URL。
- Agent 工具必须注册和限定，不允许自由访问内部系统。

### 11.7 敏感日志

禁止记录：

```text
密码
JWT / Token
MinIO Access Key / Secret
AI Service Key
Qwen API Key
生产数据库凭据
图片 base64
完整文档正文
身份证号等敏感明文
```

## 12. 可观测性、异常与审计设计

> **本章总结：** 可观测性是业务正确性的一部分，而不是附加日志。每个 HTTP 请求都有 requestId，每个长任务有任务状态和阶段日志，每个 Python 调用有外部调用日志，每个项目关键操作有操作审计。系统遵循“原始错误优先、日志失败附加、成功调用缺日志不算成功”的原则，并通过依赖健康接口分别呈现 MySQL、Redis 和 MinIO 的状态与错误原因。

### 12.1 追踪模型

```mermaid
flowchart LR
    RID[requestId] --> HTTP[HTTP访问日志]
    RID --> AUDIT[操作审计]
    RID --> EXT[外部调用日志]
    HTTP --> TASK[taskId]
    TASK --> STAGE[阶段日志]
    EXT --> TRACE[providerTraceId]
    TASK --> BIZ[bizType + bizId]
```

### 12.2 HTTP 日志

`RequestIdFilter`：

- 接收或生成 `X-Request-Id`。
- 写入 MDC。
- 响应头和响应体返回 requestId。
- 记录 method、path、status、costMs。

### 12.3 操作审计

关键项目操作至少记录：

```text
projectId
operatorId
action
objectType
objectId
requestId
ipAddress
detail JSON
createdAt
```

当前已重点覆盖项目创建、更新、删除、状态和设置变更；其余上传、下载、审查、报告、OCR、权限变更应持续补齐。

### 12.4 外部调用日志

每次 Python/AI 调用记录：

```text
serviceName
callType
projectId
requestId
requestSummary
responseSummary
status
costMs
errorMessage
```

处理原则：

- Python 失败时保留 Python 原始业务错误。
- 若同时发生日志写入失败，把日志失败作为 suppressed/附加诊断。
- Python 成功但必需的外部调用日志写入失败时，不能返回成功。

### 12.5 依赖健康

`GET /api/system/dependencies/health` 分别检查：

```text
MySQL
Redis
MinIO
```

返回单项 `status`、`elapsedMs`、`errorMessage`，整体可为 `UP` 或 `DEGRADED`，不得只抛出笼统异常。

### 12.6 异常策略

| 异常 | 处理 |
| --- | --- |
| 参数/枚举错误 | 明确指出参数问题 |
| 无权限 | 不暴露资源详情 |
| 资源缺失 | `40400` |
| 状态竞争 | `40900` |
| 文件/解析失败 | 保存失败状态和原因，允许合规重试 |
| 模型/RAG/OCR 失败 | 保存调用日志和业务失败，不生成假结果 |
| 持久化失败 | 回滚事务或执行对象存储补偿 |
| 队列投递失败 | Outbox 记录次数、错误和下次时间 |
| Worker 非 owner | 冲突失败，不允许完成任务 |

## 14. 性能、容量与可用性设计

> **本章总结：** 普通查询应保持秒级响应，模型、文档、索引和报告等耗时流程通过异步或专用超时处理。系统通过分页、最大行数、文档长度、块数、模型调用数、任务租约、Outbox 重试和文件大小限制控制资源放大。需求文档中的准确率和时延是验收目标，不应在没有生产压测与样本评估的情况下视为已经达到。

### 14.1 目标指标

| 指标 | 目标 |
| --- | --- |
| 普通接口平均响应 | ≤ 3 秒 |
| 普通接口 P95 | ≤ 5 秒 |
| 系统月可用率 | ≥ 99.5% |
| 问答准确率目标 | ≥ 90% |
| 审查准确率目标 | ≥ 95% |
| OCR 准确率目标 | ≥ 95% |
| 标准报告目标 | ≤ 20 秒 |
| 复杂报告目标 | ≤ 30 秒 |

这些是目标值，需通过真实样本、模型额度、网络条件和压测验证。

### 14.2 资源保护

| 风险 | 保护措施 |
| --- | --- |
| 大文件上传 | Multipart 与业务层双重大小限制 |
| 超长文档 | `max-document-chars` 显式失败 |
| 单次模型上下文过大 | 分批解析、Markdown 分块 |
| 审查调用爆炸 | 最大块数和 `REVIEW_MAX_MODEL_CALLS` |
| 数据库结果过大 | 最大 100 行默认、SQL timeout |
| 列表扫描 | 分页、索引、项目过滤 |
| 队列积压 | 任务统计、Outbox 重试、Worker 横向扩容 |
| 模型服务抖动 | connect/read timeout、有限重试、显式失败 |

### 14.3 扩展策略

- Java 应用尽量保持无状态，认证状态依赖 JWT/Redis，业务状态在 MySQL。
- Worker 使用数据库 claim，可多实例竞争。
- Python 服务可独立扩容。
- MinIO 和向量库按容量独立扩展。
- 当单模块成为明显瓶颈时，可按现有应用服务边界拆分；拆分前先消除跨模块 Mapper 依赖。

### 14.4 缓存策略

当前 Redis 主要用于认证安全、队列和通用缓存基础。后续缓存应遵循：

- 仅缓存可重建数据。
- Key 必须使用 `smart-worksite:` 命名空间。
- 项目级缓存必须包含 `projectId`。
- 权限、项目状态和资源状态变化时及时失效。
- 不缓存明文密码、Token 或敏感完整结果。

## 16. 现状差距、风险与演进路线

> **本章总结：** 当前系统已经形成完整工程骨架和多个业务闭环，但距离生产级平台仍有明确差距。最高优先级不是继续堆叠页面，而是统一 AI 边界与异步任务、补齐安全和可观测性、完成生产部署与真实样本评估。演进应以“不破坏现有契约、迁移可回滚、状态可追踪”为原则，逐步减少本地异步和直接模型耦合。

### 16.1 高优先级技术债

| 优先级 | 问题 | 风险 | 建议 |
| --- | --- | --- | --- |
| P0 | Java 文件解析仍直连 Qwen-VL | 密钥边界和 AI 责任边界不一致 | 新增 Python 文档解析 API，迁移适配器和配置 |
| P0 | OCR/文件解析未统一 Outbox | 进程退出丢任务、无租约和 owner | 接入统一任务 Worker |
| P0 | 默认开发凭据与密钥默认值 | 生产误用风险 | 生产启动校验、密钥管理、禁用默认账号 |
| P0 | 生产部署与监控未成体系 | 可用性与故障定位不足 | 增加网关、监控、告警、备份和运行手册 |
| P1 | 审计覆盖仍不完整 | 操作留痕缺口 | 补齐上传、下载、权限、OCR、报告等审计 |
| P1 | PDF 解析/OCR 预处理不统一 | 扫描件能力不稳定 | Python 侧文本/图片分流和逐页合并 |
| P1 | 报告 PDF/预览/分享未完成 | 产品闭环不完整 | 基于版本与文件模块扩展 |
| P1 | 系统配置 API、公告、维护模式未完成 | 运维控制不足 | 在 `system_config` 基础上实现受控配置 |
| P2 | 文件去重、版本、引用和清洗不足 | 数据治理能力有限 | 分阶段增加引用表/版本和清洗任务 |

### 16.2 演进阶段

#### 阶段 A：边界收敛

- Python 新增文档解析端点。
- Java 移除 Qwen Key 直接配置。
- `FILE_PARSE`、`OCR_RECOGNITION` 纳入 Outbox Worker。
- 统一 system-safe 文件访问和任务失败写入。

#### 阶段 B：生产可靠性

- 部署 Nginx、Java、Python、MySQL、Redis、MinIO、向量库。
- 增加 Prometheus/OpenTelemetry 或等效指标与链路。
- 建立任务积压、失败率、模型耗时、存储容量告警。
- 完成备份恢复演练和密钥轮换。

#### 阶段 C：业务完善

- 报告 PDF、预览、版本、归档和分享。
- 审查结果导出、人工复核和统计。
- 文件版本、去重、引用关系和清洗。
- 数据源表/字段白名单与敏感字段策略。

#### 阶段 D：智能质量治理

- 建立问答、审查、OCR、报告评测集。
- 记录模型、Prompt、知识版本与成本。
- 引入离线评估、灰度模型切换和质量回归。
- 完善 Agent 工具审计和多步骤流程。

### 16.3 微服务拆分条件

只有当出现以下明确需求时再拆分：

- 模块需要独立扩缩容或发布节奏。
- 单体资源隔离不足。
- 团队边界稳定且接口契约成熟。
- 事务可被 Saga/事件一致性替代。
- 监控、服务发现、配置和安全基础已具备。

推荐拆分顺序通常为：

```text
Python AI Service（已独立）
→ 文件/对象服务
→ 任务执行服务
→ 知识/RAG 编排服务
→ 报告/审查执行服务
```

拆分前必须确保业务模块不直接访问其他模块 Mapper。

## 17. 验收与运维检查清单

> **本章总结：** 系统验收不能只检查页面“能打开”，而应验证权限、项目隔离、状态机、外部失败、数据落库、任务恢复和审计证据。运维检查则必须覆盖依赖、队列、租约、存储、模型服务、默认凭据和备份。以下清单可作为上线前评审的最小基线。

### 17.1 功能验收

- [ ] 登录、退出、禁用用户旧 Token 失效。
- [ ] 平台管理员与普通用户项目列表隔离。
- [ ] DISABLED/ARCHIVED 项目拒绝业务写入。
- [ ] 文件上传失败时无孤儿元数据或尽力清理对象。
- [ ] 模板变量/规则按真实文件顺序解析。
- [ ] 知识文档仅从 `PENDING/FAILED` 提交索引。
- [ ] QA 跨项目知识库/数据源在 AI 调用前失败。
- [ ] 审查单块无证据不会直接判定全文不符合。
- [ ] 报告失败重试保留成功变量。
- [ ] OCR 发票类型和自定义字段校验有效。
- [ ] 政策文章按 URL Hash 去重并进入项目知识库。
- [ ] 任务终态不能取消，失败任务受重试上限控制。

### 17.2 数据与可靠性验收

- [ ] Flyway `V1`—`V19` 连续执行。
- [ ] 33 张表结构和索引存在。
- [ ] 关键 insert 生成 ID，update 检查影响行数。
- [ ] Outbox 投递失败记录错误和下一次时间。
- [ ] Worker 非 owner 无法完成任务。
- [ ] 长文档审查调用前续租并检查取消。
- [ ] 外部调用成功但日志写入失败时业务不返回成功。
- [ ] MinIO、Redis、MySQL 单项故障在健康接口中可定位。

### 17.3 安全验收

- [ ] 生产 JWT Secret 不使用默认值。
- [ ] 默认管理员密码已修改或账户已禁用。
- [ ] Qwen Key 只存在 Python 环境。
- [ ] 数据源密码为 `AES_GCM:` 密文。
- [ ] MinIO Bucket 私有，URL 有效期受控。
- [ ] 日志无密码、Token、Key、base64 和身份证号明文。
- [ ] 数据库问答只读账号、SQL 校验、超时和最大行数有效。

### 17.4 上线运维

- [ ] 前端 History 回退已配置。
- [ ] Python `/v1/health` 和 Java 依赖健康可监控。
- [ ] Outbox Dispatcher 和 Worker 开关符合环境。
- [ ] Worker ID 唯一，租约时间合理。
- [ ] MySQL、MinIO、向量库备份完成并验证恢复。
- [ ] 任务积压、失败率、模型耗时和容量告警已配置。
- [ ] 模型额度、超时与降级提示已验证。

### 18.1 术语

## 结语

智慧工地大模型应用系统的核心价值，不只是调用模型生成文字，而是把项目权限、文件资料、知识检索、业务数据、智能判断、长任务、结果文件和审计证据串成可运行的工程闭环。当前系统已经具备模块化单体、独立 Python AI 服务、项目隔离、可靠任务和多类智能应用的基础；后续建设应优先收敛 AI 与任务边界、强化生产可观测性和安全，再在稳定底座上扩展 PDF 报告、完整数据治理和模型质量评估。
